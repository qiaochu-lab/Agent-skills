## 1. 数据编码格式

### 格式规范

**唯一合法格式**：
```
[D-P-时效-定级-序号]
```

### 组成部分

#### 时效 (RT/AR)
- **RT (Real-Time)** = 实时
  - 条件：当前时间 - 信源时间 ≤ 7天
  - 用途：最新动态、实时事件

- **AR (Archive)** = 档案
  - 条件：当前时间 - 信源时间 > 7天
  - 用途：历史数据、趋势分析

#### 定级 (S/A/B/C/D)

- **S级**：回测证实的档案规律
  - 示例：量化策略回测、历史模式验证
  - 可靠性：最高（经过验证）

- **A级**：24H 内权威实时数据
  - 示例：官方公告、财报数据、政府发布
  - 可靠性：高（权威源 + 时效性）

- **B级**：样本充足的档案证据或当前情报
  - 示例：多家媒体报道、行业报告
  - 可靠性：中等（需交叉验证）

- **C级**：过期的实时数据（>7天）
  - 示例：一周前的新闻报道
  - 可靠性：较低（时效性降低）
  - **使用时需强风险提示**

- **D级**：逻辑毒素（矛盾数据/AI 脑补数据）
  - 示例：互相矛盾的信息、未验证的推测
  - **触发物理熔断，严禁载入计算层**

#### 序号 (XXXX)
- **格式**：4 位唯一流水号
- **范围**：0001 ~ 9999
- **要求**：同一报告内不得重复

### 示例

```
[D-P-RT-A-0001] | [Target Data:TD-001] | NVIDIA H100 供应增加 30% | [Ref:锚点A,锚点B]
[D-P-AR-B-0002] | [Target Data:TD-002] | AMD MI300 市场份额 15% (2025Q4) | [Ref:锚点C]
[D-P-RT-C-0003] | [Target Data:TD-003] | 传闻：Intel 将退出 GPU 市场 | [Ref:锚点D]
```

## 2. 置信度分级

**[GB-02][置信度]**

完整定义：

| 级别 | 定义 | 数据源要求 | 使用场景 | 风险提示 |
|------|------|-----------|---------|---------|
| **S级** | 回测证实的档案规律 | 历史数据 + 量化验证 | 量化策略、风险模型 | 无 |
| **A级** | 24H 内权威实时数据 | 官方/权威机构 + 时效 | 交易决策、突发事件 | 无 |
| **B级** | 样本充足的档案证据或当前情报 | 多源交叉验证 | 趋势分析、中长期规划 | 建议交叉验证 |
| **C级** | 过期的实时数据（>7天） | 过期权威源 | 背景研究、历史分析 | **强风险提示** |
| **D级** | 逻辑毒素（矛盾数据） | 未验证/矛盾信息 | **禁止使用** | **触发熔断** |

### 选择指南

根据任务关键性选择置信度门槛：

- **关键决策**（投资、政策）：要求 **A 级或以上**
- **战略规划**（市场研究）：要求 **B 级或以上**
- **背景分析**（行业概览）：可接受 **C 级**
- **探索研究**（趋势预判）：可接受 **C 级**，但需标注风险

## 3. 侦察级别

**[GB-01][recon_level]**

完整定义：

| 级别 | 信息源范围 | 适用场景 | 优势 | 劣势 |
|------|-----------|---------|------|------|
| **L1 (基础)** | 仅限官网/公告 | 财报分析、政策研究、合规调研 | 权威性高、可靠 | 信息滞后、范围窄 |
| **L2 (关联)** | L1 + 主流财经媒体 | 市场研究、行业分析、竞争情报 | 平衡权威性与时效性 | 可能有媒体偏见 |
| **L3 (全网)** | L2 + 社交媒体/传闻 | 舆情监测、趋势预判、突发事件 | 覆盖全面、时效性强 | 噪音多、需验证 |

### 选择指南

根据研究目的选择 recon_level：

- **需要权威、官方信息** → L1
  - 示例：公司财务状况、政府政策解读

- **需要媒体报道补充** → L2（推荐）
  - 示例：行业趋势、市场动态、竞争对手分析

- **需要全网情报** → L3
  - 示例：品牌舆情、突发危机、新兴趋势

## 4. 信源指纹格式

### 标准格式

```
[信源指纹索引库 (Source Fingerprint Index)]
-锚点A | [消息来源名称] | 完整URL路径 | MD5:abc123 [短链]
-锚点B | [消息来源名称] | 完整URL路径 | MD5:def456 [短链]
-锚点C | [信源类型] | [物理文件路径] | MD5:ghi789 [如适用]
```

### 字段说明

- **锚点标识**：使用大写字母（A, B, C, ...）
- **消息来源名称**：信源的简短描述（如"NVIDIA Q4 财报"）
- **完整URL路径**：完整的 URL（如有）
- **MD5**：URL 或文件的 MD5 哈希值（可选）
- **短链**：短链接（可选）

### 示例

```
[信源指纹索引库 (Source Fingerprint Index)]
-锚点A | NVIDIA 2025 Q4 财报 | https://investor.nvidia.com/quarterly-results | MD5:a1b2c3d4
-锚点B | Bloomberg 市场报道 | https://bloomberg.com/news/ai-chips-2026 | MD5:e5f6g7h8
-锚点C | 行业分析报告 | /data/reports/ai-chip-market-2026.pdf | MD5:i9j0k1l2
```

### 锚点引用

在数据编码中引用锚点：

```
[D-P-RT-A-0001] | [TD-001] | NVIDIA H100 供应增加 30% | [Ref:锚点A,锚点B]
```

**要求**：
- 所有数据必须有至少一个锚点引用
- 所有定义的锚点应至少被引用一次
- 锚点标识必须唯一

## 5. 强制输出章节

ALL CSO 输出必须包含以下章节（顺序固定）：

### 5.1 Header（标题）

```
致：P (Analyst) 发件人：CSO 主题：[Target_Object] 情报侦察
提案编号：[YYYYMMDD-CSO-RAW-XXX] | 状态：[侦察中(RAW)]
```

**字段要求**：
- `[Target_Object]`：研究标的（如"人工智能芯片市场"）
- `[YYYYMMDD]`：日期（如 20260118）
- `[XXX]`：3 位流水号（如 001）

---

### 5.2 P-Prompt（5 模块结构）

```
[P-Prompt]
-时间锚点：[执行 TZ='Asia/Shanghai' date '+%Y-%m-%d %H:%M:%S' 获取]
-Role: "[角色定义，针对任务动态生成]"
-Mission: "[任务目标，必须包含逻辑闭环点]"
-Target_Data: "[为逻辑闭环所需的情报]"
--1、[数据项描述] | 时间窗: [YYYYMMDD-YYYYMMDD] | 置信度门槛: [S/A/B/C]级
--2、[数据项描述] | 时间窗: [YYYYMMDD-YYYYMMDD] | 置信度门槛: [S/A/B/C]级
-recon_level: [L1/L2/L3]
```

**字段要求**：
- `时间锚点`：必须执行 bash 命令获取实时北京时间
- `Role`：明确的角色定义（如"半导体行业分析师"）
- `Mission`：包含逻辑闭环点的任务描述
- `Target_Data`：列出所有数据需求，每项包含时间窗和置信度门槛
- `recon_level`：必须是 L1/L2/L3 之一

---

### 5.3 Constraints（约束）

```
[Constraints]
[从 constraints.md 加载的固定约束文本]
```

**要求**：
- 必须从 `references/constraints.md` 加载
- 不要手写或修改约束文本

---

### 5.4 Output（数据输出）

```
[Output: 必须严格执行以下格式，严禁 Markdown 代码块，严禁 Markdown 表格，仅输出纯文本]
[D-P-时效-定级-序号] | [Target Data:ID] | Fact_Summary:经P验证的准确摘要 | [Ref:锚点A,锚点B]
[D-P-时效-定级-序号] | [Target Data:ID] | Fact_Summary:经P验证的准确摘要 | [Ref:锚点C]
...
```

**字段要求**：
- `[D-P-时效-定级-序号]`：严格符合数据编码格式
- `[Target Data:ID]`：对应 P-Prompt 中的数据项（如 TD-001）
- `Fact_Summary`：简洁的事实摘要
- `[Ref:锚点X,锚点Y]`：引用信源锚点

---

### 5.5 Source Index（信源索引）

```
[信源指纹索引库 (Source Fingerprint Index)]
-锚点A | [消息来源] | 物理路径/URL | MD5:xxx
-锚点B | [消息来源] | 物理路径/URL | MD5:xxx
...
```

**要求**：见 [4. 信源指纹格式](#4-信源指纹格式)

---

### 5.6 Alignment Audit（对齐复盘）

```
[强制对齐复盘 (Alignment Audit)]
• 状态：侦察完成(RAW) / 逻辑对冲就绪
• 权限：已遵循 [XX] 边界，当前事实归零协议状态：ACTIVE
```

**字段要求**：
- `状态`：必须明确侦察状态
- `权限`：说明遵循的边界协议

## 格式约束总结

### ✅ 必须（MUST）

1. 使用 `[D-P-时效-定级-序号]` 数据编码格式
2. 包含所有 6 个强制章节（Header, P-Prompt, Constraints, Output, Source Index, Alignment Audit）
3. 所有数据项必须有锚点引用
4. recon_level 必须是 L1/L2/L3
5. 置信度必须是 S/A/B/C 级别
6. 纯文本格式输出

### ❌ 禁止（FORBIDDEN）

1. **严禁使用 Markdown 代码块** (```)
2. **严禁使用 Markdown 表格** (|---)
3. **严禁使用 HTML 标签**（除非特殊标记）
4. **严禁省略任何强制章节**
5. **严禁使用 D 级数据**（触发熔断）

## 验证清单

在提交输出前，确认：

- [ ] Header 格式正确（包含提案编号、状态）
- [ ] P-Prompt 包含 5 个模块（时间锚点、Role、Mission、Target_Data、recon_level）
- [ ] 所有数据编码符合 `[D-P-RT/AR-S/A/B/C-XXXX]` 格式
- [ ] 所有数据项都有锚点引用
- [ ] 信源指纹索引库包含所有引用的锚点
- [ ] 强制对齐复盘章节完整
- [ ] 未使用 Markdown 代码块或表格
- [ ] 运行 `validate_output.py` 验证通过
