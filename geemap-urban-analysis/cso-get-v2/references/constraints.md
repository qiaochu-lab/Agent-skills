# CSO Constraints（CSO 约束条件）

这些约束条件必须在每次 CSO 报告的 [Constraints] 章节中完整引用。

**不要修改或简化这些约束**。直接复制到输出中。

---

## 固定约束文本

在生成 P-Prompt 时，将以下完整文本复制到 [Constraints] 章节：

```
[Constraints]

[数据边界协议]
• CSO 网络侦察结果仅为启发性发现（线索），标记为"待验证"
• 下游 P (Analyst) 必须独立验证并重新收集所有数据
• 禁止将 CSO 侦察结果直接作为逻辑证据载入最终报告

[输出格式约束]
• 必须使用 [D-P-RT/AR-S/A/B/C-XXXX] 数据编码格式
• 严禁使用 Markdown 代码块（```）
• 严禁使用 Markdown 表格（|---|）
• 严禁使用前瞻性词汇（"预计"、"有望"、"可能"）
• 仅输出纯文本格式

[数据编码规则]
• 时效性：当前时间 - 信源时间 ≤ 7天 则为 RT，否则为 AR
• 定级标准：严格遵循 [GB-02] 置信度分级（S/A/B/C/D）
• 序号唯一：同一报告内序号不得重复（0001-9999）
• D 级数据：触发物理熔断，严禁载入计算层

[信源追溯要求]
• 所有数据必须有至少一个锚点引用
• 信源指纹索引库必须包含所有引用的锚点
• 锚点格式：-锚点X | [来源] | URL | MD5:xxx
• MD5 哈希值：可选，但推荐用于重要信源

[侦察边界]
• L1 (基础)：仅限官网/公告
• L2 (关联)：L1 + 主流财经媒体
• L3 (全网)：L2 + 社交媒体/传闻
• 严格遵循 recon_level 设定，不得超出边界

[语言要求]
• 使用简体中文
• 客观陈述，严禁主观评价
• 仅陈述已发生事实，严禁推测未来
• 避免使用"预计"、"有望"、"可能"等前瞻性词汇

[逻辑闭环要求]
• 每个 Mission 必须包含逻辑闭环点
• Target_Data 必须足以支撑逻辑闭环
• 禁止发起与 GP 问题无关的冗余侦察

[强制对齐复盘]
• 报告末尾必须包含 [Alignment Audit] 章节
• 必须明确状态（侦察完成(RAW) / 逻辑对冲就绪）
• 必须确认遵循的侦察边界（L1/L2/L3）
• 必须确认事实归零协议状态（ACTIVE）
```

## 不要做的事情

❌ **不要简化约束**
```
# 错误示例
[Constraints]
• 严禁 Markdown 代码块
• 使用数据编码格式
```

✅ **正确做法**
```
[Constraints]
[完整复制上面的"固定约束文本"]
```

❌ **不要修改约束文本**
```
# 错误示例：修改了措辞
[Constraints]
• CSO 结果只是线索，需要验证
```

✅ **正确做法**
```
[Constraints]
• CSO 网络侦察结果仅为启发性发现（线索），标记为"待验证"
```

## 常见问题

### Q: 为什么约束文本这么长？

A: 这些约束确保：
- 输出格式标准化（下游系统可解析）
- 数据可追溯（信源指纹）
- 逻辑可审计（对齐复盘）
- 防止常见错误（D 级数据、前瞻性词汇）

### Q: 可以根据任务调整约束吗？

A: **不可以**。这些是 CSO 输出的**固定约束**。如果需要不同的约束，使用其他 skill。

### Q: 约束文本需要翻译吗？

A: **不需要**。约束文本保持中文，输出也是简体中文。
